## Real issue: `retarget_last_result` will need updating for struct opcodes

When you add:

```txt
NEW_STRUCT
STRUCT_GET_CONST
```

remember to add them to `retarget_last_result`.

Otherwise later source lowering like:

```kiln
e := Entity {}
x := e.hp
```

may still work via `MOVE`, but you’ll miss the existing retarget optimization pattern.

Shape:

```odin
case .LOAD_CONST, .LOAD_FUNC, .MOVE, .NEW_ARRAY, .NEW_MAP, .NEW_STRUCT, .NEG, .NOT, ...
```

and:

```odin
case .INDEX_GET, .ARRAY_GET_CONST, .MAP_GET_CONST, .STRUCT_GET_CONST, ...
```

Not Phase 1 critical if no parser syntax yet, but as soon as source syntax lands, this is easy to forget.

## Real issue: const-field ops should decide whether bad const payload is bytecode bug or runtime error

Current const-key ops do this kind of thing:

```odin
key_object := cast(^StringObject)const_pool[int(inst.c)].(^Object)
```

For compiler-generated bytecode, that is fine: the compiler owns the invariant.

But your Phase 1 plan says:

```txt
const must be string object
```

If you literally want `STRUCT_GET_CONST` to check that, then it should validate and runtime-error.

If you treat bad const payload as malformed bytecode, it can assert/panic/cast directly.

Either is fine, but lock it. I’d personally say:

```txt
const payload type is a bytecode invariant, not user runtime behavior
```

Then no defensive check. The receiver/field existence/value check are runtime behavior.

## Real issue: `ARRAY_PUSH` currently panics on non-array

This is okay today because parser only emits `ARRAY_PUSH` for array literals immediately after `NEW_ARRAY`.

But once you’re adding new VM-only struct proof paths, remember the distinction:

```txt
internal construction opcodes may trust bytecode invariants
source-visible/user-directed ops should runtime-error
```

So `NEW_STRUCT` can trust its const pool def if that is a bytecode invariant. `STRUCT_SET_CONST` should runtime-error for wrong receiver/wrong assigned value because those come from runtime values.

## My main warning for Phase 1

Don’t turn bytecode invariants into runtime guards everywhere.

For struct opcodes, I’d divide it like this:

```txt
Bytecode invariants:
    NEW_STRUCT const operand is StructDefObject
    STRUCT_GET_CONST / SET_CONST field-name operand is StringObject

Runtime checks:
    receiver is struct
    field exists
    assigned value is not nil
    assigned value matches field template
    struct field assigned value has exact def identity
```

That keeps the VM direct without hiding malformed compiler output behind “safe” fallbacks.
